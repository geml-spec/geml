# 查一条调用链,不该让 AI 把半个仓库 grep 一遍

*三行命令,把整个代码库的调用图摊成一棵纯文本 GEML 文档树:每个方法一个 `#id` 块,`#calls` / `#called-by` 双向可查——它调谁、谁调它,一秒看穿。而且能 grep、能 diff、自带版本历史,不绑数据库。*

AI 拍着胸脯说:「没别的地方调它了,放心改。」

上线第二天,炸了。

那是个前后端一个仓库的全栈项目。我要改一个函数的签名,顺手让 agent 帮我摸一遍「谁在调它」。它 grep 了函数名,翻出三个调用点,读了几个文件,很笃定地告诉我:就这三处,改完对齐一下就行。于是我改了,本地测试全绿,提交,发布。第二天线上报错——还有第四个调用者,藏在一个接口后面,名字对不上、grep 抓不到,是运行时才分发过去的。agent 没看见它,因为它压根**看不见**:对一个 agent 来说,你的代码库就是一堆文本文件。它 grep 它能看见的,然后信心十足地漏掉它看不见的。

那一下午我不是在改代码,我是在替 agent 补它本该有、却没有的那张图。

## 到底缺了什么

其实缺的东西很具体:**agent 手里没有一张它能拿得动、查得动的调用图。**

你会说,IDE 不是有「查找所有引用」「调用层级」吗?有。但它是一个锁死在你编辑器里的重量级索引:大 monorepo 上转半天,跨不过你那个仓库里 TS 调 Java 的边界,而且——最要命的——**你的 agent 用不了它**。它活在一个 GUI 后面,不是一段 agent 读得到的文本。Sourcegraph 那一类更强,但那是一整套数据库加服务器:不是你 agent 能随手 grep 的东西,不跟着你的代码一起进版本库,过两周就和代码对不上了。

于是每次追一条调用链、估一次「改这儿会炸哪儿」,agent 都得从零来过:grep 一个名字,把一大片文件拉进上下文找感觉,烧掉成千上万 token,拼出一张一半是对的心智图——然后漏掉动态分发、漏掉隔壁包里那个调用者,给你一个自信但残缺的答案。

缺的,是一张**就是文本**的调用图:可寻址、可校验、带版本——正是 GEML 给文档做过的那件事,只不过这回对象是代码。所以我给它做了一张。三行命令:

```sh
npm i -g @geml/geml         # 需要 Node 22+
geml codemap build          # --root 默认当前目录:自动认语言 → 索引 → 一张合并图落到 ./.geml-code-graph/
geml codemap serve          # 浏览器自动打开这张图
```

- **TS/JS**——零配置,`build` 自己把 scip 索引器拉下来。
- **Java / C / Python / Go / Kotlin**——多下一个 [Joern](https://docs.joern.io/installation),解压后把那个目录喂给 build(或丢进 PATH 省掉这一步)。
- 前后端混在一个仓库?全部合并成**一张图**。

## 它为什么是文本,不是数据库——这才是关键

你整个代码库的调用图,就是一棵 GEML 文档树:每个源目录一份文档,每个方法一个 `#id` 空块(块上的 `src=` 直接指到源码第几行到第几行),配至多三张边表——`#calls`(它调谁)、`#called-by`(谁调它)、`#unresolved`(连抽取器自己都没解析出的盲区)。

于是那三个最日常的动作,都退化成一句 `geml get`:

```sh
# 它调谁(追下游,排障用)——顺着 doc.geml#id 引用一路往下跟
geml get .geml-code-graph/server.c.geml '#processCommand'
geml get .geml-code-graph/server.c.geml '#calls'

# 谁调它(追上游,估爆炸半径)——连「文件:行」调用点都列给你
geml get .geml-code-graph/server.c.geml '#called-by'

# 一个符号到底在哪儿定义——名字直接查到锚点
node -e "console.log(require('./.geml-code-graph/_index/name-lookup.json')['processCommand'])"
```

而这正是最妙的地方:因为它是纯文本、可寻址的块,**你的 agent 用的还是那套 `geml get #id`——和它改文档时用的一模一样**。没有新 API 要学,没有数据库要连,没有服务器要起。它 grep 得动、diff 得出、`.gemlhistory` 存得下历史。开头那个漏掉的第四个调用者?现在它白纸黑字躺在 `#called-by` 表里,连在哪一行调的都写着。

## 它扛得住规模

「文本能扛几十万条边?」能——正因为图是*数据表*,不是每个节点一个文件。

拿 valkey 这种真实项目实测:**14,406 个符号,摊成 44 份文档**——不是一万四千个碎文件,文件数少了约 300 倍,寻址粒度却一点没降,`geml get doc.geml '#符号'` 照样精确到单个方法。把 Joern 的精确解析拉起来:9,192 个方法、66,543 个调用点、**23,235 条跨文件已解析调用**,`verify` 亚秒跑完,浏览器秒开。几万源文件、几十万条边,照样瞬间打开、随手查。

## 而且它不会骗你,也不会过期

- **边全都真校验。** `geml codemap verify` 把任何一条解析不到的边变成构建失败(exit 1)。图一旦红了,就是它过期了——先重建,再信它导航。断链**可检测**是特性,不是缺陷。
- **自带历史。** build 加 `--history`,变更的文档快照进各自的 `.gemlhistory`;`geml history log` 看这张图是怎么随代码长出来的,`geml revert doc '#某方法' --to -1` 把单个方法的调用关系回滚一版。
- **永不脱节。** 挂一个 commit hook,每次你改代码,它在后台自动刷新——代码和图一起 push,图永远追得上代码。
- **它本身就是一种图格式。** 一行 `=== diagram {format=geml-code-graph src=.geml-code-graph/index.geml} ===`,就把整张调用图嵌进任意一份 GEML 文档——设计文档里活的架构图,不再是手画完就烂掉的截图。

## 把丑话说前头

你就该对一个新东西带着怀疑,所以我把短板摆明:

- **盲区它自己标出来,不假装看全了。** `#unresolved` 表里是抽取器没能解析的调用——**那是盲区,不是「没有」**:函数指针、动态分发、反射,都会落这儿。它如实列给 agent 看,让你知道哪几处该退回去 grep,而不是给你一个漏了却装作完整的答案。这份诚实,恰恰是你敢信它其余部分的理由。
- **精度分级,不吹。** C/C++/Java/JS/Python/Rust 是一梯队(`resolution: cpg`,高置信);Go/Ruby/Kotlin 这些先过冒烟测试再用,报告里如实标。碰上虚分发、接口多实现,它给你一整个候选集,而不是硬收敛成一个「看起来对」的答案——**那一组才是答案,别只信第一个**。
- **早,且刻意做小。** 跟着 `@geml/geml` 包一起发,想接进 agent 还有一层 MCP 封装(`geml codemap mcp`);不用它,光 CLI 那条路也照样跑通。

## 最唬人的,其实不是这个图

说到底,这套东西最值得琢磨的地方,不是「又一个代码图工具」。

而是——**我没给 GEML 加任何一条新语法。** 整张调用图,就是那三件套:可寻址的块 + 可校验的引用 + 版本历史,原封不动套在代码上。同一个原语,昨天摊开一份满是表格图表的技术方案,今天摊开你整个代码库。

代码图是工程师最熟、又最难搞的场景之一——如果一个纯文本原语连这个都啃得下来,那它到底能有多能打,你心里大概就有数了。这才是我拿它做代码图的真正原因:它是一块最硬的试金石。

## 来试,顺便验我的数

别信我这些数字,自己跑一遍最快:

- **进你自己最大最乱那个仓库**,`npm i -g @geml/geml`,然后 `geml codemap build && geml codemap serve`——看它几秒钟摊出整张调用图,悬停一个方法,它的整条调用链亮起来,点一下节点,源码就贴在图旁边。
- **不想装?** playground 里就挂着一张**这个仓库自己的**活调用图,浏览器直接点:https://geml-spec.github.io/geml/playground/ ——顺手弄断一个引用,看构建变红。
- **仓库和规范:** https://github.com/geml-spec/geml ——欢迎提 issue、拍砖,也欢迎拿第三种语言写个 adapter 接进来。

我做这张图,是因为一个看不见调用链的 agent,把我坑到线上炸了一回。我想听听:你现在让 agent 追调用链、估爆炸半径,都是怎么搞的?这套东西,又会在你哪儿撑不住?
