# 种子帖(geml-code-graph)—— 中文

> 工程师传播点。比「文档格式」更容易在技术社区炸开:每个用 coding agent 的人都被「agent 追调用链靠 grep 瞎猜」坑过。发帖仍守《种子帖须知》——先讲问题、最多一个链接、坦白自己做的、真心求拍砖并一定回评论、按社区错峰。

## 发帖须知(承接主种子帖,codegraph 版加两条)

- **先讲问题再讲项目。** 「agent 拍胸脯说没别处调它,结果线上炸了」比「geml-code-graph 是什么」勾人得多。
- **一个链接:** 优先 playground 里那张**活的、这个仓库自己的**调用图——点开即玩,比截图有说服力。
- **主打「文本 vs 数据库」这个反差**,别陷进「和 Sourcegraph / IDE 比功能」——那是拿短处比长处。卖点是:它是你 agent 能 grep、能 diff、进版本库、永不脱节的**文本**。
- **诚实是护城河:** 主动说盲区(`#unresolved`:函数指针 / 动态分发解析不了)。技术社区吃这套,反感「全知全能」的吹。

## V2EX / 掘金 风格帖

**标题(选一个):**

- 我让 agent 摸一遍「谁调了这个函数」,它 grep 完拍胸脯说没别处——上线第二天炸了
- agent 追调用链是笔隐形税:它把半个仓库 grep 进上下文,还是漏了接口后面那个调用者
- 分享:三行命令,把整个代码库的调用图变成纯文本(agent 能 grep、能 diff、自带版本)

**正文:**

> 全栈仓库,改一个函数签名,让 Claude Code 先摸一遍谁在调它。它 grep 出三个调用点,很笃定地说就这仨。我改了,本地全绿,发布——第二天线上报错:还有第四个调用者,藏在接口后面运行时才分发,名字对不上、grep 抓不到。agent 没看见,因为对它来说你的代码库就是一堆文本文件,它 grep 它能看见的,漏掉它看不见的。
>
> 根子:agent 手里没一张它拿得动、查得动的调用图。IDE 的「查找引用」锁在编辑器里、跨不过 TS↔Java、而且 agent 用不了;Sourcegraph 那类是数据库+服务器,不进你版本库、两周就过期。于是我把调用图做成了纯文本——`geml codemap build` 三行命令,整库调用图摊成一棵 GEML 文档树:每个方法一个 `#id` 块,`#calls`(它调谁)/`#called-by`(谁调它,连文件:行都给)双向可查,agent 用的还是那套 `geml get #id`。valkey 实测 14,406 符号→44 份文档(不是一万四千个碎文件),Joern 精确解析下 23,235 条跨文件已解析调用,`verify` 亚秒。
>
> 关键是它诚实:解析不了的调用(函数指针/动态分发)进 `#unresolved` 表如实标出来——是盲区不是「没有」,让你知道哪儿该退回去 grep。而且断链 = 构建失败,自带 `.gemlhistory` 历史,commit hook 自动刷新永不脱节。TS/JS 零配置,Java/C/Py/Go/Kt 加个 Joern。
>
> playground 里挂着一张**这个仓库自己的**活调用图,浏览器直接点:https://geml-spec.github.io/geml/playground/
>
> 真心想听:你们现在让 agent 追调用链、估「改这会炸哪儿」,都咋搞的?这套在哪儿会不成立?

## r/LocalLLaMA / r/LLMDevs 角度(token / 上下文)

上下文那批人,把钩子拧到 token 上:

> agent 追一条调用链,本质是在跟上下文窗口打仗:grep 一个名字 → 把一大片文件拉进上下文 → 烧几千 token 拼一张一半对的图 → 还漏了动态分发。我把整个代码库的调用图预先算成纯文本 GEML,每个方法一个 `#id` 块、`#calls`/`#called-by` 双向表——agent 不再 slurp 整个文件,`geml get #id` 只取它要的那一块(和它改文档时同一个原语)。盲区进 `#unresolved` 如实标,不装看全。有 MCP 封装能直接接进 agent。浏览器活 demo,免装:https://geml-spec.github.io/geml/playground/ —— 想听听你们现在怎么把「代码库结构」喂给 agent 的。
